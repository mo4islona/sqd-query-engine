# Storage Architecture

## Why Not RocksDB

The previous storage layer (`sqd_storage`) uses a custom columnar page format on top of RocksDB.
RocksDB is a general-purpose LSM-tree key-value store designed for random writes, point lookups,
and transactional updates. Blockchain data has none of these access patterns — it is append-only,
immutable, queried in ranges, and deleted in bulk. This mismatch causes fundamental problems:

### Access Pattern Mismatch

| Capability | RocksDB Designed For | Blockchain Data Actually Does |
|-----------|---------------------|-------------------------------|
| Writes | Random key updates | Sequential appends (time-ordered UUIDs, monotonic page indices) |
| Reads | Point lookups by key | Range scans by block number |
| Deletes | Single key delete | Bulk delete entire datasets or block ranges |
| Compaction | Merge overlapping key ranges | sqd_storage has its own chunk merge (1.9x WA target), RocksDB compaction runs on top |
| Transactions | ACID across keys | Used for optimistic conflict detection, but data is never updated |

Note: sqd_storage writes are technically sequential (UUID v7 keys + monotonic page indices),
which is favorable for RocksDB's LSM tree. The mismatch is not in write ordering but in the
unnecessary overhead layers: LSM compaction on top of sqd_storage's own compaction, shared
column families preventing isolation, and optimistic transactions with infinite retry loops.

### Current Problems (Validated Against Code)

| Problem | Root Cause (code evidence) | Impact |
|---------|---------------------------|--------|
| **Can't measure dataset size** | All data in shared `CF_TABLES` CF. No size metadata in `Chunk` struct. Must iterate all table pages to compute. | No per-dataset storage accounting |
| **Can't set per-dataset quota** | No isolation — 5 global CFs (`CF_DATASETS`, `CF_CHUNKS`, `CF_TABLES`, `CF_DIRTY_TABLES`, `CF_DELETED_TABLES`) shared by all datasets | One runaway dataset fills disk for all |
| **Slow dataset deletion** | Cleanup iterates pages with `cursor.seek()/next()` and calls individual `batch.delete_cf()` per key — NOT `DeleteRange()`. Space reclaimed after RocksDB compaction. | Deletion of large dataset takes minutes + compaction |
| **Write stalls possible** | `OptimisticTransactionDB` with **infinite retry, no backoff** on commit conflicts. No `max_write_buffer_number` or L0 stall tuning. 8MB flush threshold. | Under contention, writers spin. Under load, L0 can stall all writes |
| **Shared memory pressure** | Single 256MB block cache + single memtable for `CF_TABLES` across all 200 datasets. No per-dataset memory control. | Unpredictable eviction, one hot dataset thrashes cache for all |
| **Layered write amplification** | sqd_storage chunk merge: ~1.9x (configurable). RocksDB LSM compaction on top: ~5-10x with default level compaction. Combined: ~10-20x total. | Elevated disk I/O, SSD wear |
| **Build complexity** | RocksDB C library with jemalloc feature flag, platform-specific linking | CI/CD friction, allocator conflicts |
| **Minimal RocksDB tuning** | No bloom filters, no block size tuning, no compaction style config, default level multiplier | Suboptimal for columnar page workload |

### Proposed Design Benefits

| Metric | Current (sqd_storage) | Proposed (WAL + Parquet) | Improvement |
|--------|----------------------|--------------------------|-------------|
| **Write amplification** | ~10-20x (sqd_storage merge 1.9x × RocksDB LSM ~5-10x) | 2x (write IPC once, compact to parquet once) | **5-10x less disk I/O** |
| **Write blocking** | Possible (L0 stalls, optimistic tx retries) | Never (file append + atomic rename) | **Zero write stalls** |
| **Storage size** | 252 MB per EVM chunk | 70 MB (parquet) | **3.6x smaller** on disk |
| **Query throughput (cold)** | 196 rps (evm/usdc, CPU=30) | 767 rps (parquet) | **3.9x faster** |
| **Query throughput (hot)** | 196 rps (evm/usdc, CPU=30) | 768 rps (spillover) | **3.9x faster** |
| **Dataset size tracking** | Requires iterating all table pages | File size sum | **Trivial** |
| **Dataset quota** | Not possible | Check before write/compact | **Built-in** |
| **Dataset deletion** | Iterate + individual key deletes + compaction | `rm -rf` | **Instant** |
| **Block range eviction** | Same as deletion (iterate + delete + compact) | Delete chunk directory | **Instant** |
| **Memory per dataset** | Shared 256MB block cache + shared memtable | ~0 (OS page cache) | **Predictable** |
| **200 datasets** | All share 5 CFs, fight for cache and compaction | Independent directories | **No interference** |
| **Dependencies** | RocksDB C lib + jemalloc | Pure Rust (arrow-rs) | **Simpler builds** |
| **Crash recovery** | RocksDB WAL replay + sqd_storage dirty table cleanup | Skip incomplete IPC tail | **Simpler** |

### Why Not Fix sqd_storage Instead?

1. **Layered compaction is structural**: sqd_storage has its own chunk merge (1.9x WA target), and RocksDB runs its own LSM compaction on top. Two compaction layers on immutable data that never needs merging.

2. **Shared CFs require shared resources**: Moving to per-dataset CFs means 200 × memtable (min 4MB each = 800MB). `WriteBufferManager` can cap total but then each dataset gets ~1MB memtable — frequent flushes, more L0 pressure.

3. **Deletion is inherently slow**: Code uses individual `batch.delete_cf()` per page key (not even `DeleteRange`). Fixing to `DeleteRange` helps but space is still reclaimed only after compaction.

4. **Optimistic transactions add overhead**: `OptimisticTransactionDB` with snapshot isolation and infinite retry on conflicts. Blockchain writes don't need transactions — blocks are written atomically by one producer per dataset.

The proposed design eliminates RocksDB entirely, using only Arrow IPC (for fast writes)
and Parquet (for fast reads) — both are native Rust, zero-copy, and purpose-built for
columnar analytics.

## Requirements

The query engine serves as a **hot cache / staging node** for blockchain data.
- 231 datasets (131 EVM, 88 Substrate, 10 Solana-family, 2 Hyperliquid)
- Blocks arrive every 50-400ms per dataset (up to 4000 blocks/sec total)
- Writes must NEVER be blocked
- Blocks must be queryable **immediately** (per-block head updates, validated on live portal)
- Forks/reorgs require rollback support
- Cold storage on separate machines; local data evicted on cold storage confirmation
- Most queries target chain tip ("latest" blocks)

## Architecture

```
Block arrives (per dataset, every 50-400ms)
     ↓
[In-memory buffer]              ← queryable immediately, head updates per block
     + [IPC crash log]          ← append-only file, durability only (never queried)
     ↓  memory cap exceeded (configurable, default 500MB)
[Spillover parquet]             ← small parquet files for unfinalized overflow
     ↓  finalization signal
[Compacted parquet chunks]      ← sorted, merged, query-optimized
     ↓  eviction (cold storage confirms)
[deleted]
```

### How legacy does it (validated against live portal)

Legacy uses `MaybeOnHead` event: when the data source stream ends (no more blocks
available), it **unconditionally flushes** the buffer to RocksDB — even for 1 block.
The 200K rows / 30MB threshold is only for mid-stream overflow protection.

Verified: Solana head advances every ~400ms, Ethereum every ~12s on `portal.sqd.dev`.

Our equivalent: block arrives → push to in-memory buffer → head visible instantly.
Background thread flushes finalized blocks to parquet.

### Forks, Reorgs, and Finalization

- **In-memory buffer** = mutable layer. Supports instant truncation on reorg.
- **Parquet** = immutable layer. Only finalized blocks compacted here.

**Reorg handling:**
```
Fork detected (block_number <= current head):
  1. Truncate in-memory buffer to fork point    (instant)
  2. Truncate crash.ipc to matching offset      (ftruncate, ~μs)
  3. If reorg crosses parquet boundary:         (rare)
     - Delete affected spillover parquet files
  4. Resume ingestion from fork point
```

### Directory Layout

```
{dataset_id}/
  crash.ipc                    ← IPC crash log (append-only, durability only)
  chunks/
    24550597-24550820/         ← compacted parquet chunk, sorted
      blocks.parquet
      transactions.parquet
      logs.parquet
    24550821-24551044/
      ...
```

No `active/` directory — in-memory buffer IS the active data.
`crash.ipc` is never queried during normal operation (only on crash recovery).

## Write Path

```
Block arrives for dataset X:
  1. If reorg (block_number <= current head):
     a. Truncate in-memory buffer to fork point
     b. Truncate crash.ipc to matching offset
  2. Push RecordBatches to in-memory ring buffer       (~μs, zero-copy)
  3. Append same batches to crash.ipc                  (~μs, sequential write)
  4. Head visible to queries immediately
```

## Flush to Parquet (background, non-blocking)

```
Trigger: finalization signal received
  1. Take snapshot of finalized blocks from buffer
  2. Sort by table sort keys (metadata-driven)
  3. Write chunks/{first}-{last}/*.parquet
  4. Remove flushed blocks from buffer
  5. Truncate crash.ipc (remove flushed prefix)
```

Flush does NOT block writes — new blocks continue appending to buffer.

**Spillover** (when memory cap exceeded, background, non-blocking):

Only needed for chains with large finalization gaps (Arbitrum, zkSync, Celo).
Most chains (180+ of 231) never trigger spillover.

```
Example: Arbitrum, 8394 unfinalized blocks, 500MB memory cap

Buffer: [block 100] [block 101] ... [block 8494]
         ↑ oldest                        ↑ head (latest)

Buffer (4.2 GB) > cap (500 MB) → spillover triggers:

  1. Take oldest 200 blocks (100-299) from buffer
  2. Write parquet: spillover-100-299/*.parquet            (~30-100ms)
  3. Remove blocks 100-299 from memory
  4. Now: memory = blocks 300-8494, spillover = blocks 100-299 on disk
  5. Repeat until buffer ≤ cap

Result:
  memory:    blocks 7495-8494 (~500 MB, last ~4 min)      → queries ~μs
  spillover: blocks 100-7494 (parquet files on disk)       → queries ~1-2ms

  Query for block 8490 (latest) → memory → instant
  Query for block 5000 (older)  → spillover parquet → 1-2ms (still fast)
```

Spillover parquet files are regular parquet, read by the same `ParquetChunkReader`.
On reorg: delete affected spillover files (rare — most reorgs are 1-2 blocks deep,
within memory buffer).

For Arbitrum: spillover ~100ms every ~4 min = 0.04% CPU.
For most chains (gap 0-3): **spillover never triggers**.

## Query Path

### CompositeChunkReader

A `DatasetReader` wraps all tiers and implements `ChunkReader` — transparent to
the rest of the pipeline. No changes to `execute_chunk()`, joins, or output assembly.

```rust
struct DatasetReader {
    memory: MemoryChunkReader,           // latest N blocks in RAM
    spillover: Vec<ParquetChunkReader>,  // spillover parquet files (unfinalized overflow)
    chunks: Vec<ParquetChunkReader>,     // compacted parquet chunks (finalized, sorted)
}

impl ChunkReader for DatasetReader {
    fn scan(&self, table: &str, request: &ScanRequest) -> Result<Vec<RecordBatch>> {
        let mut batches = Vec::new();
        batches.extend(self.memory.scan(table, request)?);
        for s in &self.spillover {
            batches.extend(s.scan(table, request)?);
        }
        for c in &self.chunks {
            batches.extend(c.scan(table, request)?);
        }
        Ok(batches)
    }
}
```

### Why this works without pipeline changes

1. `execute_chunk()` takes `&dyn ChunkReader` — doesn't care about data source
2. `scan()` returns `Vec<RecordBatch>` — pipeline already handles multiple batches
   (parquet reader returns one per row group)
3. Joins via `KeyFilter` use `HashSet<Vec<u8>>` — work across batches from any tier
4. Output assembly groups by `block_number` — blocks from different tiers merge naturally
5. Within-block sort by `item_order_keys` — already implemented

### Block range routing (optimization)

`DatasetReader` skips tiers that can't contain requested blocks:

```
Query from_block=8000, to_block=8494:
  → chunks (blocks 100-7494):       SKIP (to_block < from_block)
  → spillover (blocks 7495-7999):   SKIP (to_block < from_block)
  → memory (blocks 8000-8494):      SCAN → [batch1]
  → return [batch1]
```

"Latest" queries hit **only memory** — zero disk I/O.

### Example: full range query

```
Query: "USDC transfers, blocks 100-8494"

DatasetReader.scan("logs", request):
  → chunks/100-500/:     ParquetChunkReader → [batch1, batch2]  (stats skip non-matching RGs)
  → spillover 501-7494:  ParquetChunkReader → [batch3, batch4]  (stats skip non-matching RGs)
  → memory 7495-8494:    MemoryChunkReader  → [batch5]          (filter in memory)
  → return [batch1, batch2, batch3, batch4, batch5]

execute_chunk() processes 5 batches as usual:
  → predicate filter → KeyFilter join → group by block_number → JSON output
```

IPC crash log is **never queried** during normal operation.

### When memory isn't enough (spillover to disk)

Per-dataset configurable memory cap (default 500 MB). When exceeded, oldest
blocks spill to small parquet files on disk. Same `ParquetChunkReader` reads them.

| Source | Query latency | When used |
|--------|-------------|-----------|
| In-memory buffer | **~0.01 ms** (μs) | Normal: latest blocks in RAM |
| Spillover parquet | **~1-2 ms** | Memory cap exceeded |
| Compacted parquet | **~1-2 ms** | Historical queries |

The 1-2ms disk penalty is still **10-100x faster** than current sqd_storage
(which always reads from RocksDB = disk).

## Eviction

```
Cold storage: "dataset X has blocks up to 1399"
  → rm -rf chunks/24550597-24550820/
  → rm -rf chunks/24550821-24551044/
```

Block range encoded in directory name. Deletion = `rm -rf`. Instant.

## Dataset Management

- **Size**: sum of chunk file sizes + crash.ipc (tracked in memory)
- **Quota**: check before parquet flush
- **Delete dataset**: `rm -rf {dataset_id}/`
- **200 datasets**: independent directories, no shared state

## Crash Recovery

```
On startup:
  1. Scan crash.ipc → rebuild in-memory buffer (skip incomplete tail)
  2. Scan chunks/ → rebuild chunk index
  3. Resume ingestion from buffer head + 1
```

## Implementation

Implemented in `src/scan/`:

| Module | Purpose |
|--------|---------|
| `memory_backend.rs` | `MemoryChunkReader` — in-memory buffer, per-block RecordBatches |
| `composite_reader.rs` | `CompositeChunkReader` — merges tiers with block range routing |
| `crash_log.rs` | IPC append-only crash log writer + recovery reader |
| `parquet_writer.rs` | Sorted parquet flush with 8K row groups |
| `dataset_store.rs` | Orchestrates buffer + crash log + chunks per dataset |

All tiers implement the existing `ChunkReader` trait — no changes to the query
engine pipeline (`execute_chunk`, joins, output assembly).

## Appendix: Per-Block Memory Cost Analysis

Measured on R2 production data (EVM: 224 blocks, Solana: 48 blocks).

| Table | Parquet/block | Arrow/block | Note |
|-------|--------------|-------------|------|
| blocks | 0.5 KB | 1.4 KB | Tiny |
| logs | 44 KB | 500 KB | data, topics |
| statediffs | 76 KB | 550 KB | address, prev, next |
| traces | 122 KB | **2,145 KB** | call_input dominates (1.1 MB) |
| transactions | 70 KB | 522 KB | input (305 KB) |
| **Total** | **311 KB** | **3,718 KB** | **11.7x** expansion |

Parquet→Arrow expansion is 11.7x, driven by variable-length hex strings.
Binary encoding (hex → raw bytes) saves ~50% on hex columns.
